Advanced XML: An introduction

Splunk highly recommends starting with simple XML - only move to advanced XML if you cannot achieve what you want with simple XML. Review the simple XML examples to see how to build various types of dashboards and forms using simple XML.

Advanced XML is useful for specifying the functional wiring of the modules in the view. In other words, advanced XML is really dealing with how the data flows between modules. Advanced XML also determines the visual layout through layoutPanel attributes, but layout is really a secondary function of advanced XML.

Learning advanced XML

There are over a hundred modules in Splunk's advanced XML. From Splunk Web, you can navigate to a full specification here:

http://localhost:8000//modules

You can also access the module reference from Splunk core documentation here: Module Reference.

Viewing source files

All but a few of the examples in this app contain links to the XML code for the example. However, you can also view the XML source of any view by specifying the showsource REST parameter (?showsource=1) to any view URL. For example:

  http://localhost:8000/en-US/app/dashboard_examples/advanced_intro1?showsource=1

This provides a treeview of the modules in your view plus the actual XML source.

Note: The XML source strips whitespace and comments, so it might be better to view the XML source from the links in the apps or in the actual source files.

Editing the source for the examples

You can make a clone of any view by going to manager > User interface > Views, and then clicking clone on the view you want to copy, and then modify.

Splunk suggests you copy examples to the local directory before making any edits. Use the local directory to make sure updates to the Dashboard Examples app do not over write your edits. Navigate to:

$SPLUNK_HOME/etc/apps/dashboard_examples/default/data/ui/views

Copy the example you want to modify and place it in the local directory. Edit the view in an XML editor of your choice.

$SPLUNK_HOME/etc/apps/dashboard_examples/local/data/ui/views

After editing files on disk, refresh the modules to view your changes. For example:

http://localhost:8000/en-US/debug/refresh

When editing nested modules, keep your indentation accurate. Use an XML aware editor to help you track opening and closing tags. On Linux platforms you may find tidy a useful timesaver:

  tidy -xml -q -i -wrap 0 my_view.xml > pretty_view.xml
