HTTP Response Times For splunkd

This view exposes high response times for HTTP requests serviced by the splunkd back-end server.

In the most common scenario, the initiator/client for these requests is Splunk Web, the front-end HTTP server powering the graphical user interface that users directly interact with. Of course, other clients such as the Splunk Enterprise CLI or SDKs can be the initiators of these requests.

For more information on the splunkd and Splunk Web processes, refer to this documentation topic.

These HTTP requests served by splunkd target endpoints of the REST API, as highlighted in the diagram below:

By that definition, the response times measured here are for requests between Splunk Web and splunkd, not between the client's browser and Splunk Web.

The primary source of events for the searches in this view are the HTTP access logs recorded by splunkd in $SPLUNK_HOME/var/log/splunk/splunkd_access.log.

Some of the typical questions this view can answer are:

  • Are there any REST API endpoints that are taking an abnormally long time to respond to requests?

  • Are those high response times responsible for incidents where Splunk Web will report that the splunkd daemon timed out?

  • Can these high response times be correlated with external factors such as high system load or search concurrency?

High response times over time

This chart shows occurrences of high response times (1 second or more) to REST API requests served by splunkd, aggregated over time using three different statistical functions: median, 90th percentile and maximum. The dotted red line labeled "Splunk Web timeout threshold" is drawn at the 30-second mark. This represents the response time threshold above which Splunk Web shows the user an error, usually a "500 - internal server error" indicating a read time out from splunkd.
